# Git Commit Convention (Conventional Commits)

Tài liệu này quy định chuẩn viết commit message cho dự án nhằm đảm bảo tính nhất quán, dễ theo dõi lịch sử và hỗ trợ tự động hóa (ví dụ: CHANGELOG).

---

## 1. Cấu trúc cơ bản

Mỗi commit message nên tuân thủ cấu trúc: **Header**, **Body** (tùy chọn) và **Footer** (tùy chọn).

```text
<type>(<scope>): <subject>

<body>

<footer>
```

- `Header`: bắt buộc
- `Body`: tùy chọn (giải thích thêm)
- `Footer`: tùy chọn (Breaking Changes, liên kết issue, ... )

---

## 2. Các loại Commit (`type`)

Sử dụng các tiền tố sau để phân loại thay đổi:

| type   | Ý nghĩa | Ví dụ |
|--------|---------|--------|
| `feat` | Một tính năng mới (Feature) | `feat(api): add jwt authentication flow` |
| `fix`  | Sửa lỗi (Bug fix) | `fix(ui): resolve overlap issue on mobile devices` |
| `docs` | Thay đổi về tài liệu (README, Wiki, ...) | `docs: update deployment guide for production` |
| `style`| Thay đổi về format code (khoảng trắng, dấu phẩy...) - không đổi logic | `style: format code according to team rules` |
| `refactor` | Sửa đổi code nhưng không sửa lỗi cũng không thêm tính năng | `refactor(model): simplify user role checks` |
| `perf` | Cải thiện hiệu năng xử lý (Performance) | `perf(api): reduce query latency for dashboard` |
| `test` | Thêm hoặc sửa đổi các bộ test | `test(api): add coverage for jwt token refresh` |
| `build` | Thay đổi hệ thống build hoặc dependencies (npm, docker, maven...) | `build(docker): bump base image version` |
| `ci` | Thay đổi cấu hình CI/CD (GitHub Actions, Jenkins...) | `ci: update workflow to use node 20` |
| `chore` | Các thay đổi nhỏ khác không liên quan đến source code | `chore: update tooling config` |

Gợi ý: chọn `type` phù hợp nhất với mục tiêu chính của commit.

---

## 3. Quy tắc viết Header (`<type>(<scope>): <subject>`)

### Scope (Phạm vi)

Ghi rõ vùng ảnh hưởng, ví dụ: `auth`, `api`, `ui`, `model`.

Ví dụ header:
`feat(api): add JWT authentication flow.`

### Subject (Tiêu đề)

- Sử dụng câu mệnh lệnh, thì hiện tại (vd: `add` thay vì `added`)
- Không viết hoa chữ cái đầu tiên của tiêu đề
- Không để dấu chấm `.` ở cuối câu

Format chuẩn:
`<type>(<scope>): <subject>`

Nếu không có scope, có thể dùng:
`<type>: <subject>`

---

## 4. Thân bài & Chân trang (Body & Footer)

### Body

Giải thích **tại sao** thực hiện thay đổi này (đặc biệt khi header chưa đủ rõ).

Nguyên tắc:
- Tập trung vào “lý do” và “ngữ cảnh”
- Không nhắc lại y hệt nội dung của header

### Footer

Footer dùng để ghi chú các thông tin quan trọng như:

- `BREAKING CHANGE`: ghi rõ thay đổi gây ảnh hưởng ngược
- Liên kết tới Issue (vd: `Closes #123`)

Ví dụ Breaking Change (bắt buộc dạng `BREAKING CHANGE: ...`):
```text
BREAKING CHANGE: The data schema has changed significantly. 
Existing user records must be migrated using the provided script.
```

## 5. Ví dụ thực tế

### Thêm tính năng mới
```text
feat(api): add JWT authentication flow
```

### Sửa lỗi logic
```text
fix(ui): resolve overlap issue on mobile devices
```

### Cập nhật tài liệu
```text
docs: update deployment guide for production
```

### Thay đổi quan trọng (Breaking Change)
```text
feat(db): switch from MongoDB to PostgreSQL

BREAKING CHANGE: The data schema has changed significantly.
Existing user records must be migrated using the provided script.
```

## 6. Gợi ý (PR Template)

Nếu bạn muốn chuyên nghiệp hóa quy trình hơn nữa, bạn có thể thêm file `.github/PULL_REQUEST_TEMPLATE.md` để chuẩn hóa nội dung PR (tóm tắt thay đổi, issue liên quan, checklist, …).

Ví dụ tối giản (gợi ý):
```md
## Summary

## Related issue
- Closes #123

## Type of change
- [ ] feat
- [ ] fix
- [ ] docs
- [ ] refactor
- [ ] perf
- [ ] test

## Checklist
- [ ] Tests added/updated
- [ ] Docs updated (if needed)
```
